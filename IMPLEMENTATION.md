# LogiTrack — Shipment Priority Prediction System

## Overview

This document describes the AI-powered shipment priority prediction system that classifies shipments into **High / Medium / Low** priority based on 6 factors. The system uses a trained machine learning model (RandomForest) to automatically predict priority when a shipment is created or confirmed.

## How the Priority Model Works

### The 6 Priority Factors

The model considers these factors, ranked by importance (weight):

| # | Factor | Weight | What it means | How it's normalized |
|---|--------|--------|---------------|---------------------|
| 1 | `shipment_type` | 3.0 | Express shipments get highest priority | normal=0.0, express=1.0 |
| 2 | `distance_km` | 2.5 | Longer distance = more delay risk | `min(distance/2500, 1.0)` |
| 3 | `restrictions` | 2.0 | Fragile or cold chain = needs special handling | `count / 2.0` (0-2) |
| 4 | `time_window` | 1.5 | Morning window = tighter deadline | flexible=0.0, afternoon=0.5, morning=1.0 |
| 5 | `volume_score` | 1.0 | Larger packages = more complex logistics | `(package_base + weight/50) / 12` capped at 1.0 |
| 6 | `route_saturation` | 0.8 | Busy routes = higher congestion risk | simulated 0.0-1.0 |

### Scoring Formula

```
total_score = Σ(normalized_factor × weight) / Σ(all weights)
```

### Classification Thresholds

| Score | Priority | Meaning |
|-------|----------|---------|
| > 0.65 | **Alta** (High) | Likely to be delayed, needs immediate attention |
| > 0.35 | **Media** (Medium) | Standard priority, moderate risk |
| <= 0.35 | **Baja** (Low) | Low risk, can be processed normally |

### Volume Score Calculation

| Package type | Base size |
|-------------|-----------|
| envelope | 1 |
| box | 5 |
| pallet | 15 |

```
volume_score = base_size + (weight_kg / 2.0)
normalized   = min(volume_score / 25.0, 1.0)
```

### Distance Calculation

Distance is computed using the **Haversine formula** between Argentine province centroids. The coordinates for all 23 provinces + CABA are defined in `dataset.py`.

### Route Saturation

Simulated deterministically based on the hash of the origin-destination route pair. This gives consistent values for the same route while varying between different routes.

## Step-by-Step Setup Guide

### Prerequisites

- Python 3.10+ with pip
- Go 1.21+
- Node.js 18+ with npm

### Step 1: Install Python Dependencies

```bash
cd logitrack_ml
pip install -r requirements.txt
```

This installs: flask, flask-cors, scikit-learn, numpy, pandas.

### Step 2: Train the Model

```bash
cd logitrack_ml
python train.py
```

This script:
1. Generates 2000 synthetic shipment samples with random combinations of features
2. Labels each sample using the weighted scoring formula from `config.py`
3. Trains a RandomForestClassifier (100 trees, max depth 10)
4. Shows cross-validation accuracy (~83-84%)
5. Runs two test predictions to verify the model works
6. Saves the trained model to `model.pkl`

Expected output:
```
============================================================
LogiTrack ML — Shipment Priority Model Training
============================================================

[1/3] Generating synthetic dataset (2000 samples)...
  Priority distribution:
     alta:   297 (14.8%)
    media:   968 (48.4%)
     baja:   735 (36.8%)

[2/3] Training RandomForest classifier...
Cross-validation accuracy: 0.839 (+/- 0.010)

[3/3] Saving model...
Model saved to model.pkl

--- Sample predictions ---

  Express long-distance fragile:
    Priority: alta (confidence: 0.84, score: 0.90)
  ...

Training complete. Run 'python app.py' to start the service.
```

### Step 3: Start All Three Services

**Terminal 1 — ML Prediction Service:**
```bash
cd logitrack_ml
python app.py
```
Output: `ML Prediction Service running on :5001`

**Terminal 2 — Go Backend:**
```bash
cd logitrack_core
go run cmd/server/main.go
```
Output: `LogiTrack API running on :8080`

**Terminal 3 — React Frontend:**
```bash
cd logitrack_web
npm run dev
```
Output: `Local: http://localhost:5173`

### Step 4: Test the System

1. Open http://localhost:5173
2. Log in with: `admin` / `admin123`
3. Click **+ New Shipment**
4. Fill in the form — note the new fields:
   - **Shipment Type**: Normal or Express
   - **Time Window**: Flexible, Morning (8-12), or Afternoon (12-18)
   - **Cold Chain**: checkbox for refrigerated shipments
5. Click **Create shipment**
6. The backend will call the ML service and set the `priority` field
7. The shipment detail page shows the priority badge (red/amber/green)

## How to Change Priority Weights

To change which factors matter most (e.g., make distance #1 instead of shipment type):

### Step 1: Edit `config.py`

```python
PRIORITY_FACTORS = {
    "distance_km":      {"weight": 3.0, "description": "Longer distance = more delay risk"},  # ← Now #1
    "shipment_type":    {"weight": 2.5, "description": "Express = highest priority"},          # ← Now #2
    "restrictions":     {"weight": 2.0, "description": "Fragile/cold chain = needs special handling"},
    "time_window":      {"weight": 1.5, "description": "Morning = tighter deadline"},
    "volume_score":     {"weight": 1.0, "description": "Larger = more complex logistics"},
    "route_saturation": {"weight": 0.8, "description": "Busy route = congestion risk"},
}
```

### Step 2: Retrain the Model

```bash
cd logitrack_ml
python train.py
```

### Step 3: Restart the ML Service

Stop the running `python app.py` process and start it again. The new model will be loaded.

### Step 4: Verify

Create a new shipment and check that the priority predictions reflect the new weights.

## API Reference

### ML Service — `POST /predict`

**Request:**
```json
{
  "origin_province": "Córdoba",
  "destination_province": "Santa Cruz",
  "shipment_type": "express",
  "time_window": "morning",
  "package_type": "box",
  "weight_kg": 5.2,
  "is_fragile": true,
  "cold_chain": false
}
```

**Response (200):**
```json
{
  "priority": "alta",
  "confidence": 0.87,
  "score": 0.74,
  "factors": {
    "shipment_type":    {"value": "express",   "normalized": 1.0,  "weight": 3.0, "contribution": 0.30},
    "distance_km":      {"value": 2100,        "normalized": 0.84, "weight": 2.5, "contribution": 0.21},
    "restrictions":     {"value": 1,           "normalized": 0.5,  "weight": 2.0, "contribution": 0.10},
    "time_window":      {"value": "morning",   "normalized": 1.0,  "weight": 1.5, "contribution": 0.15},
    "volume_score":     {"value": 15.6,        "normalized": 0.31, "weight": 1.0, "contribution": 0.03},
    "route_saturation": {"value": 0.72,        "normalized": 0.72, "weight": 0.8, "contribution": 0.06}
  }
}
```

The `contribution` field shows each factor's actual impact on the final score. This makes it easy to verify: "Why was this alta? Because shipment_type (express) contributed 0.30 and distance contributed 0.21."

**Error (503):**
```json
{
  "error": "Model not loaded. Run 'python train.py' first."
}
```

### ML Service — `GET /health`

**Response (200):**
```json
{"status": "ok"}
```

### Go Backend Integration

The Go backend calls the ML service automatically when:
- A shipment is **created** (`POST /api/v1/shipments`)
- A draft is **confirmed** (`POST /api/v1/shipments/:tracking_id/confirm`)

Configuration:
- `ML_SERVICE_URL` env var (default: `http://localhost:5001`)
- If the ML service is unavailable, the shipment is created without priority (empty string). No crash.

## Graceful Degradation

If the ML service is not running:
- The Go backend logs a warning: `[MLClient] prediction service unavailable: ...`
- Shipments are created normally, with `priority` set to empty string
- The frontend handles empty priority gracefully (no badge shown)
- No errors or crashes occur

To run without the ML service, simply don't start `python app.py`. The system works normally without priority predictions.

## Retraining Workflow

After changing `config.py` weights:

```bash
cd logitrack_ml

# 1. Regenerate dataset + retrain model
python train.py

# 2. Restart the ML service to load the new model
# (stop the running process, then:)
python app.py
```

The new weights take effect immediately for all new shipments created after the restart. Existing shipments keep their original priority.

---

## Adding a New Priority Factor

1. Add the factor to `config.py` with its weight
2. Add normalization logic in `dataset.py` (for training) and `model.py` (for prediction)
3. If it's a new shipment field, add it to:
   - Go: `model/shipment.go`, `model/prediction.go`, `service/ml_client.go`
   - React: `api/shipments.ts`, `NewShipment.tsx`, `ShipmentDetail.tsx`
4. Run `python train.py` to retrain
5. Restart the ML service

## Troubleshooting

| Problem | Solution |
|---------|----------|
| `model.pkl not found` | Run `python train.py` in `logitrack_ml/` |
| ML service won't start | Check port 5001 is free, check Python dependencies are installed |
| Priority not showing in UI | Check ML service is running, check browser console for errors |
| Go backend can't reach ML | Check `ML_SERVICE_URL` env var, check ML service is running on correct port |
| Predictions seem wrong | Check `config.py` weights, retrain with `python train.py` |
| Want different priorities | Edit `config.py` weights → `python train.py` → restart ML service |
